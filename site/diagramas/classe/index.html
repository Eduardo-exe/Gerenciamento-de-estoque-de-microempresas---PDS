    Diagramas: Arquitetura de Classes e Evolução UML

    Este documento analisa a evolução do modelo de arquitetura de classes do G-Estoque, contrastando a concepção original do diagrama de classes com a implementação final no código em Python (SQLAlchemy / Pydantic / Flet).

    🔬 Evolução do Modelo: Teoria vs. Implementação Prática

    No desenvolvimento de software, é comum e esperado que os diagramas conceituais passem por um processo de refinamento arquitetural durante a transição para a linguagem programática real.

    1. O Papel do Acesso do Usuário no Diagrama Original

    Em diagramas de classes UML iniciais, muitas vezes a classe Usuario é desenhada de forma isolada do modelo de mercadorias (sem linhas ligando diretamente a produtos), pois no domínio relacional do negócio, uma conta de usuário não é dona física de uma peça. - No Código Real: No entanto, durante a execução, os serviços (services/*.py) atuam como orquestradores que recebem o usuário logado (via token JWT) e autorizam a mutação na tabela de ItemEstoque. Assim, a separação foi mantida no banco de dados (tabelas independentes para segurança e isolamento), mas unida logicamente pela camada de controle de acesso de roteadores REST.

    2. A Tabela Associativa ItemEstoque (Estoque vs. Produto)

    No modelo conceitual, um produto pode ter uma ligação com o estoque. Na modelagem relacional implementada via ORM SQLAlchemy em models/estoque.py, criamos a entidade associativa ItemEstoque intermediando Estoque e Produto. - Por que isso foi feito? Essa arquitetura em Terceira Forma Normal (3NF) permite que um produto exista no catálogo mestre com código e nome independentemente de ter saldo em prateleira, e permite que o galpão gerencie quantidades dinâmicas sem duplicar a descrição do item a cada movimentação.

    🏛️ Diagrama de Classes e Entidades do Backend (Implementação Atual)

    O diagrama Mermaid abaixo representa a estrutura de classes e modelos de dados exata implementada em nosso código de produção:

    classDiagram
    %% Modelos de Banco de Dados (ORM SQLAlchemy)
    class Usuario {
        +int id
        +string nome
        +string login
        +string senha
        +string tipo
        +boolean ativo
        +verificar_senha(senha_plana) bool
    }

    class Produto {
        +int codigo
        +string nome
        +listar_movimentacoes()
    }

    class Estoque {
        +int id
        +adicionar_item(produto, qtd)
        +remover_item(produto, qtd)
    }

    class ItemEstoque {
        +int id
        +int produto_codigo
        +int estoque_id
        +int quantidade
    }

    class Fornecedor {
        +string cnpj
        +string nome
        +string telefone
        +validar_regex_cnpj() bool
    }

    %% Schemas Pydantic (Validação de Dados API)
    class LoginRequest {
        +string login
        +string senha
    }

    class LoginResponse {
        +string access_token
        +string tipo
        +string nome
    }

    class MovimentacaoEstoqueSchema {
        +int codigo
        +string nome
        +int quantidade
    }

    class ProdutoSchema {
        +int codigo
        +string nome
        +int quantidade
    }

    class FornecedorSchema {
        +string cnpj
        +string nome
        +string telefone
    }

    %% Relacionamentos do Modelo ORM
    Estoque "1" -- "0..*" ItemEstoque : contém
    Produto "1" -- "0..*" ItemEstoque : alocado em

    %% Relacionamento com Schemas
    LoginRequest ..> Usuario : valida credenciais
    LoginResponse ..> Usuario : encapsula dados de sessão
    MovimentacaoEstoqueSchema ..> ItemEstoque : mutação de saldo
    ProdutoSchema ..> Produto : serialização JSON
    FornecedorSchema ..> Fornecedor : serialização JSON

    📐 Estrutura de Componentes do Frontend (Flet UI Classes)

    No lado da apresentação desktop em frontend/, as responsabilidades orientadas a objeto foram organizadas para separar controle de sessão de renderização gráfica:

    classDiagram
    class ApiClient {
        -string _token
        -dict _headers()
        +set_token(token)
        +clear()
        +login(login, senha) Dict
        +get_estoque() List
        +registrar_entrada(cod, nome, qtd) Dict
        +registrar_saida(cod, nome, qtd) Dict
        +cadastrar_produto_gerente(cod, nome, qtd) Dict
        +cadastrar_fornecedor_gerente(cnpj, nome, tel) Dict
        +baixar_relatorio_pdf(prefixo) str
        +get_usuarios() List
        +alterar_tipo_usuario(id, tipo) Dict
        +deletar_usuario(id) Dict
    }

    class StateController {
        -dict _state
        +get_state() dict
        +set_state(key, val)
        +clear_state()
    }

    class AppRouter {
        +main(page: Page)
        +route_change(e)
    }

    class Views {
        <<enumeration>>
        +login_view(page) View
        +estoquista_view(page) View
        +gerente_view(page) View
        +admin_view(page) View
    }

    AppRouter --> Views : empilha e renderiza (SPA)
    AppRouter --> StateController : valida permissões de acesso
    Views --> ApiClient : envia requisições HTTP REST
    Views --> StateController : lê/atualiza token e cargo
